# 功能设计：安卓应用的UI实现

## 1. 需求分析

### 1.1 UI / UX 目标
第一版的应用界面以“**极简为主，专注于核心题库与练习本身**”。
坚决避免任何不必要的功能、花里胡哨的过渡动画、社交分享与复杂的账户体系设计。用户的核心留存驱动仅靠“选取题目种类 -> 刷题 -> 反馈”这一短促有力的使用体验。

### 1.2 涵盖核心业务
- **数据包下载与管理**：直观展示哪些种类的题目包可以通过下载获取，哪些已经处于离线可用状态。
- **题目呈现排版**：全面支持 LaTeX 语法中对公式的美观、专业和响应式渲染（比如导数、极限、线性代数矩阵、概率统计计算）。
- **作答与反馈**：轻量直接的对对错判断提示界面。

---

## 2. 界面视图 (Screens) 规划

第一版 UI 将极大精简为 2～3 个核心页面：

### 2.1 主页 / 分类数据包管理页 (Home Screen)
- **展示内容**：分类项（例如：加减乘除、线性代数、导数、概率论等）通过网格或简单列表展示。
- **状态及交互展示**：
  - **未下载的类别**：显示【获取数据包/点击下载】字样及按钮（建议附带文件大小指示）。
  - **已下载的类别**：显示【开始练习】明确的入口（如果已练习一半，可追加轻微状态记录作为附加功能但不作强求）。
- **设计风格**：简单的 List / Grid 以及大面积纯色几何型设计，强调清晰可读和按钮的一目了然。

### 2.2 离线练习页 (Practice Screen)
这是核心的操作区域，整体布局从上到下应当包括：
1. **题干渲染区**：大面积极简白底/灰底区域，直接以 LaTeX 渲染公式和题目情境说明文本。
2. **选项区 / 填空区**：排列多个选项框（单选卡片等），对于数学选择题，各项内容可能也是一个 LaTeX 公式。
3. **底部控制栏**：包含【提交答案】和【下一题】等最简略的核心控件。

### 2.3 结果反馈与对齐提示 (Result Feedback)
用户提交答案后，系统给出即时反馈：
- 错误提示：以轻度红色作为背景反馈“答错了/正确的选项是 B”。
- 若题目带有解析数据，就在错误后直接通过面板滑出或弹出展开包含 LaTeX 算法推导原理的详细解析。

## 3. 技术选型与实现参考路线

对于极简应用，推荐以下强有力的前沿工具栈来确保稳定性：

### 3.1 前端 UI 响应式框架
- **原生推荐使用 Jetpack Compose**：采用最新的 Android 官方声明式 UI 框架搭建，组件模块化可高频复用，实现极简效果更加高效并且在 MVVM 架构下状态管理极为直观。

### 3.2 LaTeX 渲染方案评估
处理 LaTeX 是界面的难点。要在 Android 上完美展现出微积分或矩阵题干可以采取：
- **方案 A (轻量级/灵活性较高)**：
  在 Compose 之中嵌入 WebView 容器结合 `MathJax` 或是 `KaTeX` 对数学公式进行渲染。此法开发速度可观，且跨平台兼容性（同网页效果一样）极佳，对极其复杂公式支持得最好。
- **方案 B (纯原生渲染)**：
  借助原生库比如 `noties/Markwon` 所对应的 LaTeX 扩展或者 `JLaTeXMath-Android`，实现无网页依赖直接将字符串变为图片画布（Canvas）。适合注重无白屏刷新及较高流畅度的系统。对于需要频繁动态显示选项的情况，纯原生会有优势。

### 3.3 架构和状态管理策略
- 以单 Activity 多 Compose 页面驱动。`ViewModel` (`PracticeViewModel`) 专门负责直接读取并缓存 SQLite 下载好之后的数据包数据集合，管理目前答题状态，完成从“待答”到“提交”逻辑转换以切屏或刷题不丢失数据为最终目标。
