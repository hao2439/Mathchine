# 功能设计：题目的保存和分类

## 1. 需求分析

### 1.1 业务背景
应用需要提供多种数学题目的练习（如加减乘除、线性代数、导数、概率论等），考虑到用户可能在无网络环境下进行逻辑推断和计算，系统应当采用“强离线”的设计思路，只需一次性获取即可本地化刷题体验。

### 1.2 核心设计目标
1. **统一题目标准**：严格采用 LaTeX 格式存储与展示所有数学公示及文字内容。
2. **“数据包”机制下发与存储**：同一种类的数学题目将归档到一个专门的数据库（或打包数据模块）中。客户端通过网络下载并激活该数据包后，所有后续练习均在本地完成，无需持续的网络连接。
3. **高可扩展性**：在开发侧（或内容资源管理侧），应能快速组合并发布新的不同种类或具体题目（如新增“微积分”或单独追加几十道“线性代数”的新题）。

---

## 2. 数据组织与存储方案

### 2.1 题目数据结构模型 (Question Model)
结构化数据内部的设计示例：

```json
{
  "uuid": "linear_algebra_0001",
  "category": "线性代数",
  "question": "已知矩阵 $A = \\begin{pmatrix} 1 & 2 \\\\ 3 & 4 \\end{pmatrix}$，求其行列式 $|A|$。",
  "options": [
    "$-2$",
    "$2$",
    "$0$",
    "$10$"
  ],
  "correct_option_index": 0,
  "explanation": "$|A| = 1 \\times 4 - 2 \\times 3 = 4 - 6 = -2$"
}
```

### 2.2 开发环境管理（构建数据包）
- **原材料**：按照纯文本或结构化数据（如 JSON/CSV 或者 SQLite 导入脚本）维护在开发环境中。
- **打包方式**：通过脚本将一整种类的题目文件构建为标准数据包文件（例如预编译的 SQLite 数据库 `algebra.db` 或压缩格式包）。
- 这种独立分离使得开发者只需提供更新版的数据包，即可迅速响应内容的迭代和种类的丰富。

### 2.3 客户端存储方案选型
1. **下载模块**：App 访问统一静态资源托管（或简单接口），拉取所有已有分类目录。用户点击特定分类（如“概率论”），下载对应的数据包文件。
2. **本地入库 (SQLite / Room)**：采用 Android 原生的 SQLite 引擎来保存这些下载的数据。当数据包下载完成后，交由基于 Room 构建的 DAO 层进行索引与入库持久化。
3. **本地加载与随机抽题**：当用户开始“线性代数”练习时，只需一条从本地数据库随机取 n 条记录的 `SELECT` 语句，即时反馈，响应迅速且没有网络延迟担忧。

---

## 3. 分类扩展及维护说明
得益于离线数据包和 SQLite 强存储的灵活配合，当应用后期需要增加全新科目乃至细分知识点（例如针对考研数学特化的题库）时，应用本身的底层核心逻辑与表结构大多无需修改。只要增加新的分类索引并将新题组合成数据包发布即可。
